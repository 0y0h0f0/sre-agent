# Statuses and Public IDs

**Last updated:** 2026-06-18

Public IDs are generated by `packages.common.ids.new_id(prefix)`. Prefixes must end with `_`; the generated value is `<prefix><uuid4 hex>`, for example `inc_0f4...`.

Database integer primary keys are internal. API responses, logs, and cross-table references should use the public IDs below.

For the persistence design behind public IDs, repository transaction boundaries, checkpoint pointers, Alembic migrations, and pgvector fallback, see [Data Model, Migrations, and Persistence Deep Dive](../00-overview/data-model-migrations-persistence-deep-dive.md).
For Runbook draft publish, version creation, and amendment review/apply metadata boundaries, see [Runbook 草稿、版本与 Amendment 生命周期技术深挖](../00-overview/runbook-draft-version-amendment-lifecycle-deep-dive.md).
For Discovery run, rerun response, proposal, capability matrix, and topology boundaries, see [Discovery、Capability Matrix 与服务拓扑技术深挖](../00-overview/discovery-capability-topology-deep-dive.md).

## Severity

| Status | Meaning |
|--------|---------|
| `P1` | Highest severity / urgent page. |
| `P2` | High severity. |
| `P3` | Medium or low operational impact. |
| `P4` | Informational, resolved, or NFA-suppressed style severity. |

Provider payloads normalize common severity aliases such as `critical`, `warning`, `info`, and `resolved` into this set.

## Incident Status

Defined in `apps/api/schemas/common.py` as `IncidentStatus`.

| Status | Meaning | Typical transition |
|--------|---------|--------------------|
| `open` | Incident created and awaiting diagnosis. | Created by alert ingestion. |
| `diagnosing` | Agent diagnosis is active. | Set while a run is processing. |
| `waiting_approval` | At least one proposed action needs human approval. | Set when L2/L3 action approval is waiting. |
| `mitigated` | Mitigation succeeded but final resolution may still be pending. | Set after successful action/verification paths. |
| `resolved` | Incident is closed as resolved. | Final terminal healthy state. |
| `failed` | Incident handling failed. | Terminal error state. |

Open-fingerprint deduplication applies while status is not in `resolved`, `failed`, or `mitigated`.

## Agent Run Status

Defined as `AgentRunStatus`.

| Status | Meaning |
|--------|---------|
| `queued` | Run row exists and Celery task should execute it. |
| `running` | Worker has started processing. |
| `waiting_approval` | Graph interrupted for approval. |
| `succeeded` | Run completed successfully. |
| `failed` | Run failed; `error_code` and `error_message` should explain why. |
| `cancelled` | Run cancelled. |

Active statuses are `queued`, `running`, and `waiting_approval`. Terminal statuses are `succeeded`, `failed`, and `cancelled`.

Checkpoint fields on `agent_runs`:

| Field | Meaning |
|-------|---------|
| `checkpoint_thread_id` | Always the `agent_run_id`. |
| `checkpoint_ns` | Empty string for the current graph namespace. |
| `latest_checkpoint_id` | Latest LangGraph checkpoint pointer if available. |

`agent_runs.state` is a display/debug snapshot; it is not a replacement for LangGraph checkpoint persistence.

## Agent Node and Tool Status

Node trace records use string statuses rather than a shared enum. Current node code emits:

| Status | Meaning |
|--------|---------|
| `succeeded` | Node completed successfully. |
| `degraded` | Node completed with partial evidence or fallback behavior. |
| `failed` | Node failed. |
| `skipped` | Node intentionally skipped due to missing input or disabled capability. |

Tool calls also use string statuses. Common values include:

| Status | Meaning |
|--------|---------|
| `succeeded` | Tool call returned usable results. |
| `degraded` | Tool returned fallback/partial results. |
| `timeout` | Tool timed out. |
| `failed` | Tool failed. |
| `blocked` | Tool/action was blocked by safety policy. |

## Risk Levels

Defined as `RiskLevel`.

| Risk | Execution policy | Examples |
|------|------------------|----------|
| `L0` | Automatic, read-only. | `query_metrics`, `query_logs`, `query_traces`, `query_git`. |
| `L1` | Automatic low-risk local/system action. | `create_ticket`, `generate_report`, `warmup_cache`, `adjust_connection_pool`. |
| `L2` | Requires approval. | `restart_pod`, `restart_deployment`, `scale_deployment`, `restart_service`, `restart_statefulset`, `pause_rollout`, `resume_rollout`, `increase_memory_limit`, `scale_back`, `revert_config`. |
| `L3` | Requires approval and second confirmation. | `enable_rate_limit`, `raise_rate_limit`, `rollback_release`, `rollback_deployment`, `enable_circuit_breaker`, `switch_dns_resolver`, `failover`, `cancel_deployment`. |
| `L4` | Directly blocked; must not enter approval. | `delete_data`, `truncate_table`, `flush_cache`, `modify_database`. |

Unknown action types default conservatively to `L2`. Forbidden destructive keywords escalate to `L4`.

## Action Status

Defined as `ActionStatus`.

| Status | Meaning |
|--------|---------|
| `proposed` | Agent proposed the action. |
| `blocked` | Guardrail or execution policy blocked it. |
| `waiting_approval` | Approval record is waiting. |
| `approved` | Human approved the action. |
| `rejected` | Human rejected the action. |
| `executing` | Action execution is in progress. |
| `succeeded` | Execution succeeded. |
| `failed` | Execution failed. |

Important constraints:

- `L4` actions should become `blocked` and are never executable.
- L2/L3 execution requires an approved approval row.
- L3 execution additionally requires persisted `risk_ack=true`, `confirm_action_type`, and `confirm_target`.

## Approval Status

Defined as `ApprovalStatus`.

| Status | Meaning |
|--------|---------|
| `waiting` | Approval is awaiting decision. |
| `approved` | Approved by an operator. |
| `rejected` | Rejected by an operator. |
| `expired` | Approval expired. |

Approving or rejecting a non-`waiting` approval returns conflict behavior. Approval decisions also update the associated action status and write audit records.

## Runbook Draft, Version, and Amendment Status

| Resource | Status | Meaning |
|----------|--------|---------|
| Runbook draft | `draft` | Deterministic/template/cluster draft awaiting review. |
| Runbook draft | `pending_review` | M9 LLM-generated draft awaiting review. |
| Runbook draft | `published` | Reviewer approved the draft; a `RunbookVersion` is created. |
| Runbook draft | `rejected` | Reviewer rejected the draft. |
| Amendment draft | `pending_review` | Proposed change awaiting review. |
| Amendment draft | `approved` | Reviewer approved the amendment draft. |
| Amendment draft | `applied` | Approved amendment was recorded as applied to one target reference; current implementation stores metadata only and does not merge content. |
| Amendment draft | `rejected` | Reviewer rejected the amendment draft. |
| Amendment draft | `superseded` | Amendment was replaced by a newer amendment. |
| Runbook chunk embedding | `available` | Provider embedding is usable. |
| Runbook chunk embedding | `degraded` | Provider failed partially or fallback used. |
| Runbook chunk embedding | `failed` | Embedding generation failed. |

Draft regeneration creates a new `drf_` and sets `parent_draft_id`; it does not overwrite the original draft.

## Alert Poll Cursor State

`alert_poll_cursors` uses counters rather than a named status field:

| Field | Meaning |
|-------|---------|
| `filter_hash` | Poll configuration scope hash. |
| `fingerprint` | Alert fingerprint observed in that scope. |
| `incident_id` | Linked incident when one was created. |
| `missing_rounds` | Consecutive poll rounds where the alert was absent. |
| `first_seen_at` / `last_seen_at` | Poll observation timestamps. |

The cursor's unique key is `(filter_hash, fingerprint)`.

## Eval Status

Eval APIs expose status strings rather than a strict enum. Current code/tests commonly use:

| Status | Meaning |
|--------|---------|
| `queued` | Eval run was created/enqueued. |
| `running` | Eval execution is active. |
| `succeeded` | Eval run completed successfully. |
| `failed` | Eval run failed. |
| `completed` | Legacy/test display synonym for successful completion. |
| `pending` | Eval case has not run yet. |

CI smoke eval must use FakeLLM and enforce the documented smoke metrics.

## API Key State

API keys have booleans and timestamps rather than a status enum:

| Field | Meaning |
|-------|---------|
| `revoked` | When true, the key is rejected by middleware. |
| `expires_at` | Expired keys are rejected. |
| `last_used_at` | Best-effort touch after successful request. |
| `roles` | Coarse labels, currently stored for metadata/extension. |
| `scopes` | Granular permission tokens used by `require_scope()`. |

Bootstrap seed identity is exposed as `apik_initial` while `API_KEY_INITIAL_SEED` is accepted.

## WebSocket Event Types

Current incident WebSocket publishes JSON events for incident/node/approval updates. Common values:

| Type | Meaning |
|------|---------|
| `node_update` | Agent node status or summary changed. |
| `incident_update` | Incident-level status/details changed. |
| `approval_update` | Approval status changed or approval-related refresh was published. |
| `run_update` | Agent run status/details changed. |

Clients should tolerate unknown event types and ignore events for other run IDs.

## State Transition Rules to Preserve

When changing code that touches statuses or IDs:

1. Use existing public ID prefixes and `new_id()`.
2. Do not expose integer database IDs in public APIs.
3. Keep terminal statuses terminal unless a documented resume/rollback path exists.
4. Keep L4 actions blocked and outside approval.
5. Keep L3 second-confirmation fields exact-match validated.
6. Keep report regeneration versioned; never overwrite old report versions.
7. Keep effective config worker selection limited to `published` records.
8. Update this document when adding a new public prefix or status value.
